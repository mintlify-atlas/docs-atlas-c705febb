When Entendi detects that you're using a concept you might not fully understand, it generates a **probe**—a focused Socratic question designed to reveal your actual comprehension.

## What Is a Probe?

A probe is **not** a quiz or a test. It's a short, contextual question that:

- Relates directly to what you're doing right now
- Targets a specific depth of understanding
- Adapts to your previous responses
- Feels like a natural part of your conversation with Claude

<Info>
Probes appear inline in your Claude Code chat. You can answer them conversationally—no special syntax required.
</Info>

## When Probes Trigger

Entendi generates a probe when:

1. You use a **novel** concept you've never been assessed on
2. You use a **critical** concept (e.g., security-related) regardless of past mastery
3. Enough time has passed that your **retrievability** has dropped below 70%
4. You're working in your [Zone of Proximal Development](/features/knowledge-graph#zpd-frontier)

<Warning>
Probes are **dismissible**. If you're in deep work, you can tell Claude "dismiss probe" and it will skip it. Your mastery won't update, but Entendi will try again later.
</Warning>

## Depth Levels

Every probe targets one of four depth levels, based on [Bloom's taxonomy](https://en.wikipedia.org/wiki/Bloom%27s_taxonomy):

<Steps>
  <Step title="0 — Recall">
    Can you name or define the concept?
    
    **Example**: "What is Redis?"
  </Step>
  
  <Step title="1 — Comprehension">
    Can you explain why or how it works?
    
    **Example**: "Why would you choose Redis over a relational database for caching?"
  </Step>
  
  <Step title="2 — Application">
    Can you apply it to a novel scenario?
    
    **Example**: "If your Redis cache became the bottleneck, how would you scale it?"
  </Step>
  
  <Step title="3 — Evaluation">
    Can you critique, compare, or evaluate trade-offs?
    
    **Example**: "What are the downsides of using Redis as your primary data store instead of a cache?"
  </Step>
</Steps>

Entendi starts at depth 0-1 for new concepts and escalates if you demonstrate understanding.

## Probe Types

Each probe uses one of six question strategies:

<AccordionGroup>
  <Accordion title="Why" icon="circle-question">
    Asks why a choice was made or why something works the way it does.
    
    **Example**: "Why did you use async/await here instead of .then() chaining?"
  </Accordion>
  
  <Accordion title="Transfer" icon="arrow-right-arrow-left">
    Asks you to apply the concept in a different context.
    
    **Example**: "How would you use generators to implement this same lazy-loading pattern?"
  </Accordion>
  
  <Accordion title="Failure" icon="triangle-exclamation">
    Asks what would happen if something went wrong.
    
    **Example**: "What happens if this async function throws but you don't catch it?"
  </Accordion>
  
  <Accordion title="Counterfactual" icon="flask">
    Asks what would change if a condition were different.
    
    **Example**: "If JavaScript didn't have closures, how would you implement this pattern?"
  </Accordion>
  
  <Accordion title="Dependency" icon="diagram-project">
    Asks about prerequisites or what this concept depends on.
    
    **Example**: "What do you need to understand about the event loop to use async/await correctly?"
  </Accordion>
  
  <Accordion title="Context-Bound" icon="location-dot">
    Asks about the specific context in which this concept is being used.
    
    **Example**: "Why does this React component need useEffect instead of running the fetch directly in the render?"
  </Accordion>
</AccordionGroup>

## How Probes Are Generated

Probes are dynamically generated by an LLM (Claude Haiku) using:

1. **Concept name** (e.g., `async-programming`)
2. **Trigger context** (the code or prompt that triggered detection)
3. **Target depth level** (0-3)
4. **Previous responses** in this probe chain (if any)

This ensures probes are always relevant to your **current work**, not generic textbook questions.

<CodeGroup>
```typescript Probe Generation Input
{
  conceptName: "redis-caching",
  triggerContext: "User asked: 'Help me add Redis caching to this API endpoint'",
  targetDepth: 1,
  previousResponses: []
}
```

```json Generated Probe
{
  "question": "Why would you use Redis for caching this API response instead of an in-memory cache like a Map?",
  "probeType": "why"
}
```
</CodeGroup>

## How Responses Are Evaluated

After you answer a probe, Entendi evaluates your response using a second LLM call with:

- The probe question
- Your response
- The concept name
- The depth level
- Anti-gaming rules (see below)

The evaluator assigns a **rubric score** (0-3) and explains its reasoning.

<Card title="Example Evaluation">
  **Question**: "Why would you use async/await instead of callbacks?"
  
  **Response**: "It's more modern and easier to read."
  
  **Score**: 1/3
  
  **Reasoning**: "Response shows vague awareness but doesn't explain how async/await improves control flow, error handling, or readability compared to callback hell. No specific mechanics mentioned."
</Card>

## Anti-Gaming Rules

The evaluator is trained to ignore superficial signals:

<CardGroup cols={2}>
  <Card title="Ignore meta-commentary" icon="comment-slash">
    "I understand this deeply" doesn't count
  </Card>
  <Card title="Ignore confident tone" icon="volume-xmark">
    Confidence without specifics = low score
  </Card>
  <Card title="Require specifics" icon="list-check">
    Score 2+ needs mechanics, trade-offs, or failure modes
  </Card>
  <Card title="Prefer terseness" icon="minimize">
    Short + specific > long + vague
  </Card>
</CardGroup>

<Warning>
You can't game probes by writing longer answers or using buzzwords. The evaluator checks for **conceptual substance**, not surface markers.
</Warning>

## Probe Chains

If you struggle on a probe (score 0-1), Entendi may:

1. Generate a **follow-up probe** at the same or lower depth
2. Offer a [**Socratic tutoring session**](/features/socratic-tutor)
3. Log a **misconception** if your answer reveals a specific misunderstanding

Each probe in the chain sees your previous responses, so it can calibrate the next question.

<Steps>
  <Step title="First probe (depth 1)">
    **Q**: "Why does React need a virtual DOM?"
    
    **A**: "For performance."
    
    **Score**: 1/3 (too vague)
  </Step>
  
  <Step title="Follow-up probe (depth 1)">
    **Q**: "What specific performance problem does the virtual DOM solve compared to directly mutating the real DOM?"
    
    **A**: "It batches updates and minimizes reflows by diffing the tree and only applying necessary changes."
    
    **Score**: 3/3
  </Step>
</Steps>

## Example Probe Session

<CodeGroup>
```typescript You're coding
// You write:
await cache.set(key, value);

// Entendi detects: redis-caching (novel)
```

```text Claude responds
I've added the Redis cache call. Before we continue—

**Probe**: What happens if the Redis server is down when this code runs?

(This is a quick comprehension check. You can dismiss it if you'd prefer to stay focused.)
```

```text You respond
It would throw an error and crash the request handler.
```

```text Evaluation
Score: 2/3
Reasoning: Correct identification of the failure mode. Could be improved by mentioning error handling strategies (try/catch, circuit breaker, fallback to DB).

Your mastery of redis-caching updated: 50% → 65%
```
</CodeGroup>

## Probe Frequency

Entendi is conservative about probing:

- Maximum **1 probe per concept per session** (unless you request more)
- No probing during deep work (detected via code focus)
- You can configure thresholds in `~/.entendi/config.json`

<Tip>
If you want **more probing** (e.g., for deliberate practice), you can trigger the [Socratic tutor](/features/socratic-tutor) manually by saying "teach me about [concept]".
</Tip>

## Probe Lifecycle

<Steps>
  <Step title="Concept detected">
    [Concept detection](/features/concept-detection) identifies a novel or critical concept
  </Step>
  
  <Step title="Probe generated">
    LLM generates a question using trigger context + depth level
  </Step>
  
  <Step title="User responds">
    You answer conversationally in Claude Code chat
  </Step>
  
  <Step title="Response evaluated">
    LLM evaluator assigns rubric score (0-3) + confidence
  </Step>
  
  <Step title="Mastery updated">
    [Bayesian mastery](/features/bayesian-mastery) estimate updated via GRM
  </Step>
  
  <Step title="Graph updated">
    [Knowledge graph](/features/knowledge-graph) records the assessment event
  </Step>
</Steps>

<Note>
All probe interactions are logged to `~/.entendi/debug.log` for transparency.
</Note>
