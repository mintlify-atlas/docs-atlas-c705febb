## Overview

The `observe` tool is called whenever Claude Code detects concepts in the user's code or conversation. It:
1. Normalizes and resolves concept IDs through the three-tier pipeline
2. Builds probe candidates with Fisher information and urgency scores
3. Detects prerequisite inconsistencies and boosts those prerequisites
4. Applies codebase scoping to prioritize enrolled concepts
5. Selects the best candidate using information-theoretic ranking
6. Checks rate limits and user profile to determine intrusiveness
7. Returns probe guidance and a signed probe token

## Endpoint

```
POST /api/mcp/observe
```

## Request Body

<ParamField body="concepts" type="array" required>
  Array of concept objects to observe (1-50 items)
  
  <Expandable title="Concept object">
    <ParamField body="id" type="string" required>
      Concept identifier (1-200 chars)
    </ParamField>
    <ParamField body="source" type="enum" required>
      Source of concept detection: `package`, `ast`, or `llm`
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="triggerContext" type="string" default="">
  Brief description of what user was doing (max 1000 chars). Used to contextualize probe guidance.
</ParamField>

<ParamField body="primaryConceptId" type="string">
  The main concept from user's current work (max 200 chars). Used for conversational relevance weighting.
</ParamField>

<ParamField body="repoUrl" type="string">
  GitHub repository URL (max 500 chars). Used for codebase scoping — boosts concepts in enrolled codebases.
</ParamField>

## Response

<ResponseField name="shouldProbe" type="boolean" required>
  Whether to present a probe to the user
</ResponseField>

<ResponseField name="conceptId" type="string">
  The selected concept to probe (only if `shouldProbe=true`)
</ResponseField>

<ResponseField name="depth" type="integer">
  Probe depth: `1` (core purpose), `2` (trade-offs), or `3` (edge cases)
</ResponseField>

<ResponseField name="intrusiveness" type="enum" required>
  How to present the probe:
  - `direct`: Interrupt workflow with explicit question
  - `woven`: Weave question naturally into response
  - `skip`: Don't probe (rate limited or routine concept)
</ResponseField>

<ResponseField name="guidance" type="string">
  Context-aware guidance for Claude on what to ask (only if `shouldProbe=true`)
</ResponseField>

<ResponseField name="userProfile" type="enum" required>
  User proficiency level: `unknown`, `beginner`, `intermediate`, or `advanced`
</ResponseField>

<ResponseField name="mastery" type="integer">
  Current mastery percentage (0-100) for the selected concept
</ResponseField>

<ResponseField name="urgency" type="number">
  Probe urgency score (0-1) combining uncertainty, staleness, and Fisher info
</ResponseField>

<ResponseField name="probeToken" type="object">
  Cryptographically signed token for tamper-resistant evaluation
  
  <Expandable title="ProbeToken fields">
    <ResponseField name="tokenId" type="string">
      Unique token ID (UUID)
    </ResponseField>
    <ResponseField name="userId" type="string">
      User ID this token is valid for
    </ResponseField>
    <ResponseField name="conceptId" type="string">
      Concept ID this token is valid for
    </ResponseField>
    <ResponseField name="depth" type="integer">
      Probe depth (1-3)
    </ResponseField>
    <ResponseField name="evaluationCriteria" type="string">
      The guidance string (for auditing)
    </ResponseField>
    <ResponseField name="issuedAt" type="string">
      ISO 8601 timestamp
    </ResponseField>
    <ResponseField name="expiresAt" type="string">
      ISO 8601 timestamp (30 minutes from issuance)
    </ResponseField>
    <ResponseField name="signature" type="string">
      HMAC-SHA256 signature over all fields
    </ResponseField>
  </Expandable>
</ResponseField>

## Implementation Details

### Concept Normalization

All input concept IDs go through `resolveConcept()` which:
1. Normalizes string (lowercase, replace `_` with `-`, strip version suffixes)
2. Looks up aliases in `concept_aliases` table
3. Computes embeddings and checks for near-duplicates (cosine similarity > 0.95)
4. Returns canonical ID and `isNew` flag

Source: `src/api/lib/concept-pipeline.ts:resolveConcept` and `src/api/routes/mcp.ts:115-120`.

### Fisher Information

Fisher information measures how much we'd learn from assessing a concept:

```typescript
fisherInfo = grmFisherInformation(mu, itemParams)
```

Peaks at `mu ≈ 0` (50% mastery) where uncertainty is highest.

Source: `src/core/probabilistic-model.ts:grmFisherInformation` and `src/api/routes/mcp.ts:141`.

### Probe Urgency

Urgency combines:
- **Uncertainty**: Higher when `sigma` is large
- **Staleness**: Time since last assessment (with FSRS decay)
- **Fisher info**: Peak information gain
- **Assessment count**: Boost novel concepts

```typescript
urgency = probeUrgency({
  mu, sigma, stability,
  daysSinceAssessed,
  assessmentCount,
  fisherInfo
})
```

Source: `src/core/probe-urgency.ts` and `src/api/routes/mcp.ts:147-153`.

### Prerequisite Inconsistency Detection

If any candidate has `mu > prerequisite.mu + 0.5`, the prerequisite is added as a candidate with +0.2 urgency boost. This catches gaps where the user appears to understand a concept without mastering its prerequisites.

Source: `src/api/routes/mcp.ts:158-212`.

### Codebase Scoping

If `repoUrl` is provided and matches an enrolled codebase, all concepts in that codebase get +0.3 urgency boost.

Source: `src/api/routes/mcp.ts:214-244`.

### Candidate Selection

Uses `selectProbeCandidate()` which ranks by:
```
score = urgency * fisherInfo * (1 + relevance_boost)
```

Where `relevance_boost = 0.3 * similarity(candidate, primaryConcept)` if `primaryConceptId` is provided.

Source: `src/core/probe-selection.ts:selectProbeCandidate` and `src/api/routes/mcp.ts:246-259`.

### Rate Limiting

Checks:
1. **Minimum interval**: `now - lastProbeTime >= probeIntervalSeconds`
2. **Hourly limit**: `probesThisSession < maxProbesPerHour`

Returns `{shouldProbe: false, intrusiveness: 'skip'}` if rate limited.

Source: `src/api/routes/mcp.ts:267-283`.

### User Profiling

Computes average mastery across all assessed concepts:
- `avgMastery > 0.75` → `advanced`
- `avgMastery > 0.4` → `intermediate`
- Otherwise → `beginner`
- No assessments → `unknown`

Source: `src/api/routes/mcp.ts:285-295`.

### Intrusiveness Mapping

Novelty classification:
- **novel**: Never assessed OR `daysSince > 30`
- **adjacent**: Assessed, not stale
- **routine**: High mastery (`mu > 1.5`)

Intrusiveness by profile × novelty:

| Profile | Novel | Adjacent | Routine |
|---------|-------|----------|----------|
| unknown | direct | direct | skip |
| beginner | direct | woven | skip |
| intermediate | woven | woven | skip |
| advanced | woven | skip | skip |

Source: `src/api/routes/mcp.ts:297-309`.

### Probe Token

Generated via `createProbeToken()` with HMAC-SHA256 signature over all fields. Stored in `probe_tokens` table for server-side validation. Expires after 30 minutes.

Source: `src/core/probe-token.ts:createProbeToken` and `src/api/routes/mcp.ts:376-400`.
